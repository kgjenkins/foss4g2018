A STUDY ON NON-MERCATOR VECTOR TILING FOR AN EFFICIENT WEB MAPPING SOLUTION

Session details
Academic

A STUDY ON NON-MERCATOR VECTOR TILING FOR AN EFFICIENT WEB MAPPING SOLUTION
James Badger, Sara Saeedi, Steve Liang
Geomatics Engineering, University of Calgary, 2500 University Drive NW Calgary Canada T3A 2H4 – (jpbadger, ssaeedi, steve.liang@ucalgary.ca)

 

KEY WORDS: Vector Tiling, Web Map, Mapnik, Map Projections, OpenLayers

ABSTRACT:
In web mapping, transmitting large vector data over the Internet has been a challenging issue over the past decade. Vector tiling is a method for delivering large vector data in small pieces. Studies of traditional vector tile-based methods in web-mapping applications stayed at simple single server-client architecture with GeoJSON encoded vector tiles. However, there are still problems remained in these studies such as coordinate reference system (CRS) projection, limited scalability, and inefficient vector tile transmission. To solve these problems, a distributed memory caching implementation has been proposed and investigated based on cloud architecture for vector tile based web map applications. Furthermore, this paper explored reprojecting the vector tiles to a non-Mercator CRS and present that it is possible to combine the advantages of vector tiles with the advantages of custom map projections. This allows for the creation of more effective web maps for more communities around the world. To investigate the efficiency of the proposed approaches, a prototype of Canada arctic area vector map is developed. The results of this study show that the implemented vector-tiling solution improves the application performance and its scalability compared to naïve architecture.

1. INTRODUCTION
Vector map data are widely used in web mapping as well as Location Based Services (LBS) and mobile computing. Thus, the representation of vector data should be flexible enough to meet the efficiency requirements and be able to dynamically deliver multiple levels of vector data [2]. Vector tiles are the latest evolution of web map tiling systems, providing numerous improvements over raster tiles [1]. Unlike raster tiles, semantic information is part of the vector tiles and can be used to improve client-side rendering and functionality. For example, a client can extract custom tags and metadata from vector tiles to provide more detailed rendering and information for end users.
Sending the geospatial data with vector tiles also allows the client to decide on the rendering style. This is very powerful as it allows for client-side development of different stylesheets for different use cases, and can be used for client-side internationalization and localization.
Vector tiles also support better compression, especially at more detailed zoom levels (higher resolution) [6]. This is noticeable for users who have low bandwidth access to the internet. Vector tiles with different resolutions can clearly be used as temporary visual placeholders for other zoom levels while the appropriate zoom vector tiles are still loading and look much nicer than up-scaled raster tiles attempting the same functionality. As vector tiles work on the same tile bounding box system as raster tiles, they both can use the same web caching infrastructure [6]. This means the tiles can be cached by intermediary proxies and provide better performance to end users. It also means tiles can be individually expired and regenerated without requiring other tiles to be re-built at the same time. As tile generation is a server memory-intensive task, it is important to not to exceed server resources otherwise tiles will be slow to reach clients.
The dominant map projection system on the web is known as “Web Mercator”, a modified version of the Mercator map projection [5]. Both projections have similar advantages of being familiar to most people, well supported by both server and client-side libraries and tools, and showing the relative location of features of the world at an accuracy that is good enough for most users. On the downside, both Mercator projections have significant distortion near the Polar Regions, enough to make polar usage of the maps very degraded. This distortion also misrepresents the relative land area of regions closer to the poles relative to land areas closer to the equator, giving a false impression of the size of various continents and land masses. It is quite common in non-web cartography to use projections other than Mercator, depending on the area of interest. This is made easy by many of the Geographic Information System (GIS) applications supporting re-projection. On the web there is some support for reprojecting raster tiles, however, due to their raster nature, there is a limit to the quality of reprojection. With web vector tiles, the support is even more limited due to vector tiling being newer and less broadly supported.

2. METHODOLOGY
To solve the issue of reprojection of tiles, vector tiles can be generated in the target projection such that distortion is minimized and the correct bounding of data is provided to clients. These tiles are square tiles that cover the projected coordinate system with an x-y tile grid, which is appropriate for a 2D tiling map client.
Open source creation of vector tiles is supported by multiple projects, including the open source Mapnik (http://mapnik.org). Vector tiles are generated according to the Mapbox Vector Tile open specification [3], enabling interoperability between different vector tile generators and parsers. This open-source support forms the basis of this paper, enabling a customizable and re-usable software stack.
There are multiple documented ways to set up your own vector tile server and client stack, however, none that cover using tiles in a projection that isn’t Web Mercator (EPSG:3857) or the WGS84 coordinate system (EPSG:4326). In the first step, we have built a production system to serve raster tiles in an Arctic-focused projection, and a client library to make those tiles integrate into the popular open source mapping library, Leaflet. The lessons learned in building those systems show that non-
Mercator raster tiles are completely feasible and adaptable into a vector tile-based system.
Mapnik supports generating raster and vector geographic data in projections supported by PROJ.4 library (http://trac.osgeo.org/proj/), and can use data from multiple sources (including Shapefiles and a PostGIS database). This can be used with Tilestrata, an open source Node.js HTTP server for interfacing with Mapnik or other tile backends to serve and cache raster and vector tiles. This versatility will allow us to deploy raster and vector tiles simultaneously for comparison.
On the client side, the Mapnik vector tiles can be consumed by multiple different web map libraries, however to support the custom projection there is one choice that stands out for its support of multiple projections: OpenLayers [4]. OpenLayers (http://openlayers.org/) is a well-known open source JavaScript library for displaying map data in web browsers. This library supports advanced rendering features and client-side styling and rendering of vector tiles, both of which are under active development.

3. IMPLEMENTATION AND RESULTS
In this paper, we used open-source data from OpenStreetMap and Natural Earth Data to create tiles using Mapnik. Raster tiles will be styled using the popular openstreetmap-carto stylesheet, and vector tiles will be styled using a very simplified version of the same stylesheet. A small server is set up to host the database and shapefiles, the Tilestrata web server, and a web server to host the OpenLayers single page application. This method can be modified to serve data in other regions with projections that visualize those regions more appropriately than Web Mercator.

4. CONCLUSION
This study discussed the vector tile based web mapping system. Vector tiling was a way for delivering vector data over the web by applying the same tiling concept for traditional raster tile web mapping services. Coordinate reference systems, caches, and vector tile encoding formats played important roles in affecting the efficiency of vector tile-based mapping applications. In addition, scalability was another challenge. In this paper, we proposed a custom map projection (non-Mercator) which allows for the creation of more effective web maps for more communities around the world.
By running the tests, we have gained the following insights about the performance of the proposed solutions:
 It is possible to combine the advantages of vector tiles with the advantages of custom map projections for the arctic area.
 Vector tile based vector map overall performance can be improved by applying distributed memory caching implementation in the server side compared with naïve architecture
 Our prototype was easily scalable without losing performance based on the proposed distributed memory caching implementation
 How the vector tiles get encoded can affect tile transmission efficiency in a vector map application
